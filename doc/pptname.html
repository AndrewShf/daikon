Introduction

Instrumenting code creates a .decls file that contains program point names (PptName's) such as 
    StackAr.makeEmpty()V:::EXIT54
    StackAr.makeEmpty()V:::ENTER
    StackAr.top()Ljava/lang/Object;:::EXIT64
    StackAr.top()Ljava/lang/Object;:::EXIT65
    StackAr.top()Ljava/lang/Object;:::ENTER
This document describes the format of these PptName's. Someone writing an instrumenter for a new language must be sure to follow this format specification.

Format specification

A PptName is a string with no tabs or newlines in it. The basic format is: 
    topLevel.bottomLevel:::pptInfo
For the first example given above, the top level of the hierarchy would be StackAr, the bottom level would be makeEmpty()V, and the program point information would be EXIT54.

topLevel and bottomLevel can represent anything. Currently for Java PptName's, topLevel consists of the class name while bottomLevel consists of the method name and method signature. For C PptName's, topLevel could consist of a filename and bottomLevel could consist of a method name and signature. For IAO PptName's, topLevel could consist of an Automaton name and bottomLevel could consist of information for a transition state.

topLevel may contain any number of periods ('.'). bottomLevel and pptInfo may not contain any periods. The string ":::" may only appear once.

bottomLevel is optional. However, the period to the left of bottomLevel and the colons to the right are still requird. For example, the Java instrumenter creates an artifical object program point which contains invariants that are true for all of an object's methods, rather than for just one method: 

    StackAr.:::OBJECT
